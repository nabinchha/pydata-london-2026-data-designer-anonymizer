## Intro Slides (~12 min)

**Goal:** Motivate the problem, introduce both tools, get participants set up.

- The dataset quality problem (2 min)
  - Ad-hoc prompting produces data that lacks diversity, has no validation, can't be reproduced
  - Training/eval data deserves the same engineering rigour as models
- Data Designer overview (4 min)
  - Production pedigree: 300B+ tokens generated (as of 2026-04), instrumental in training the
    Nemotron Nano, Super, Ultra, and Omni model families -- this is the same
    tool NVIDIA uses internally for its own flagship models
  - Declarative Python framework: define what you want, not how to generate it
  - Column types: sampled, LLM-generated, code, derived, seed
  - DAG-based execution: columns reference each other, framework resolves dependencies
  - Preview (fast iteration) vs. Create (production scale)
  - Model providers: NVIDIA Build API, OpenAI, OpenRouter
- Anonymizer overview (3 min)
  - PII detection: GLiNER NER + LLM augmentation/validation
  - Four replacement strategies: substitute, redact, annotate, hash
  - Rewrite mode for full-text privacy-preserving rewrites
  - Built on Data Designer -- same ecosystem
- Setup check (3 min)
  - Confirm environment: `make setup` done, API key set (see README Quickstart)
  - Open Notebook 1 via `uv run jupyter lab`, run the import cell
  - Checkpoint: everyone sees output from import cell before proceeding

## Notebook 1: Text QA Pipeline (~35 min)

**Goal:** End-to-end exposure to every Data Designer building block by walking the
full pipeline once on a low-cost text-only example. Gets attendees confident with
the framework before we layer in multimodal complexity in Notebook 2.

**Scenario:** Build a reading-comprehension QA dataset grounded in curated
Wikipedia excerpts.

**Structure (two parts in one notebook):**

**Part A -- Controlled sampling (~12 min, no LLM calls)**
- Define sampler columns: `question_difficulty` (weighted), `question_type`
  (subcategory of difficulty), `document_type` (weighted)
- `preview(n=20)` -- inspect distributions with the helper summary table
- Discussion: sampler columns are how you control distribution *shape*; downstream
  LLM diversity follows from sampler diversity

**Part B -- LLM-grounded QA + judge (~23 min)**
- Load seed data: curated wiki excerpts parquet (checked into `data/wiki_seed.parquet`)
- `with_seed_dataset(LocalFileSeedSource(...))`
- Add LLM-generated `question` column (`LLMTextColumnConfig`, Jinja template
  references `{{ article }}`, `{{ question_difficulty }}`, `{{ question_type }}`)
- Add LLM-generated `answer` column (`LLMStructuredColumnConfig` with Pydantic
  schema, `with_trace=ALL_MESSAGES` so we can inspect generation traces)
- Add `LLMJudgeColumnConfig` scoring `faithfulness` and `completeness` on 1-5
- `preview(n=3)` -- inspect generations including judge scores
- `create(n=20)`, filter on judge scores >= 4

**Key concepts introduced:**
- `DataDesignerConfigBuilder` + `DataDesigner` runtime
- All four core column types (sampler, seed, LLM text, LLM structured, LLM judge)
- Subcategory samplers for encoding correlations between fields
- Jinja templating + automatic dependency DAG resolution
- Preview -> Create -> Filter loop
- Trace inspection for prompt debugging

**Checkpoint:** Participants have a quality-filtered QA dataset grounded in real
articles. They've seen every column type, the DAG in action, and the judge pattern
they'll re-use in Notebook 2.

## Notebook 2: Document Visual QA (~25 min)

**Goal:** Move from text to multimodal. Generate visual QA pairs from synthetic
invoice images -- the kind of data used to train/evaluate document understanding
VLMs. Same pipeline shape as Notebook 1, but with image inputs and a multimodal judge.

**Scenario:** Given synthetic invoice images generated by the bonus notebook
(`bonus_generate_invoices.ipynb`), generate summaries, structured questions, and
answers that require visual understanding of invoice fields, layout, line items,
and totals. The invoices contain realistic business identifiers (vendor names,
client names, Tax IDs, phone numbers, addresses) so Notebook 3 has a realistic
anonymization target.

**Data source:** `data/invoice_seed.parquet` (checked into the repo). Contains
base64-encoded PNG invoice images generated by Gemini via Data Designer's
`ImageColumnConfig`, plus metadata (vendor name, layout style, etc.).

**Exercises:**
1. Load seed data: invoice seed parquet with `png_base64` column + metadata
2. Add sampled columns: `question_type` (multiple choice, yes/no, short answer,
   list of items), `focus_area` (seller identity, client identity, invoice
   metadata, line items, totals and tax, cross-field, layout and structure)
3. Add a VLM-generated `summary` column: concise summary of key invoice details
4. Add a VLM-generated `question` column (`LLMStructuredColumnConfig` with
   `VQAQuestion` schema returning `text` and `rationale`)
5. Add a VLM-generated `answer` column
6. Add a boolean structured judge (`QAJudge` schema: `passes` bool + `reason`)
7. Preview with all columns including judge
8. `create(n=25)`, filter on `judge.passes == True`

**Key concepts introduced:**
- Multimodal generation (image inputs to VLM-generated columns via `multi_modal_context`)
- LLM-as-judge on multimodal outputs (builds on text judge from Notebook 1 --
  now judging visual grounding, not just text faithfulness)
- Filtering/quality gates
- The full pipeline: sample -> generate -> validate -> filter -> save

**Outcome:** Participants have a validated visual QA dataset with quality scores.
They've seen the judge pattern applied to multimodal data, and the output is ready
for anonymization in Notebook 3.

**Note:** This is the most complex notebook -- give extra time for prompt iteration
and debugging. Keep the live run small so participants can complete it in time.

## Notebook 3: Privacy with Anonymizer (~15 min)

**Goal:** The visual QA outputs from Notebook 2 contain realistic business
identifiers (vendor names, client names, Tax IDs, phone numbers, addresses,
invoice numbers). Before publishing this training data, detect and handle those
fields across all three text columns. Demonstrates anonymization as the final
pipeline stage.

**Scenario:** The VLM-generated answers, questions, and summaries from Notebook 2
contain PII-shaped identifiers from the synthetic invoices. Anonymizer detects
and replaces or redacts them.

**Exercises:**
1. Load the visual QA dataset generated by Notebook 2
2. Briefly explain package naming: dependency is `nemo-anonymizer`; import is
   `anonymizer`. Attendees already installed it via `uv sync`.
3. Configure Anonymizer: create an `AnonymizerInput` per text column (`answer`,
   `question`, `summary`)
4. `preview()` on all three columns -- inspect detected entities with Rich
   highlighting, discuss true vs. false positives
5. Try two replacement strategies side by side:
   - `Substitute` -- LLM-generated contextual replacements (names become different names)
   - `Redact` -- template-based redaction
6. Compare outputs across all three columns: which strategy preserves training
   data utility better?
7. Brief mention of `Annotate` and `Hash` strategies
8. Full `Substitute` run on all three columns, merge back onto dataframe

**Key concepts introduced:**
- PII detection pipeline (GLiNER + LLM validation)
- Replacement strategies and tradeoffs
- Preview for inspection before committing
- The composability story: Data Designer generates, Anonymizer protects

**Checkpoint:** Participants have seen PII detected and handled in their own generated
data. They understand the tradeoff between privacy and data utility.
